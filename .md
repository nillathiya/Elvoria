# Development Task List & Technical Requirements

## 1. Project Overview

Build a system with three separate roles:

1. *Admin*
2. *Peer*
3. *Normal User*

The application must use *file-based storage only*.

### Core Requirements

- Do not use MySQL, MongoDB, PostgreSQL, SQLite, or any other database.
- Store application data in structured JSON/text files.
- Use separate controllers and services for file-based data management.
- Normal users must be able to register and log in.
- Admin authentication must use a simple 6-digit numeric PIN.
- Peer authentication must use a simple 6-digit numeric PIN.
- Admin can create and manage peers.
- Peers cannot register themselves.
- Peer panel must contain only one main function:
  - Submit a TX hash.
  - Verify the TX hash directly against the blockchain.
- Once a valid TX hash is successfully submitted by a peer, permanently tie that TX hash to that peer.
- The same TX hash must never be accepted or assigned twice.
- Admin can manage multiple deposit methods.
- Admin can add multiple receiving addresses for each deposit method.
- The frontend randomly displays one configured receiving address for the selected deposit method.
- Every deposit method/network must have its own independent TX hash verification module.
- Never trust transaction details submitted by the frontend.
- All transaction information must be independently fetched and verified directly from the blockchain.

# 2. Required Roles

## 2.1 Admin

Admin has full access to the backend/admin panel.

Admin can:

- Log in using a 6-digit numeric PIN.
- Create peers.
- Edit peers.
- Enable or disable peers.
- Reset/change peer PINs.
- View peer activity.
- View TX hashes submitted by each peer.
- Manage deposit methods.
- Manage multiple receiving addresses for each deposit method.
- View all verified transactions.
- View all rejected transactions.
- Search transactions by TX hash.
- Search transactions by peer.
- View normal registered users.

Admin PIN must:

- Be exactly 6 digits.
- Contain numbers only.
- Never be stored as plain text.
- Be securely hashed before storage.

---

## 2.2 Peer

Peers are created only by the admin.

Peers cannot:

- Register themselves.
- Create other peers.
- Access admin settings.
- Access user management.
- Manage deposit addresses.
- Manage payment methods.
- Edit transactions.
- Delete transactions.

A peer can only:

1. Log in using a 6-digit numeric PIN.
2. Access their peer panel.
3. Select a supported transaction/deposit method if required.
4. Enter a TX hash.
5. Submit the TX hash.
6. Wait for direct blockchain verification.
7. See the verification result.

The peer panel should be extremely simple.

Example:

    -----------------------------------
    Peer Panel

    Transaction Method:
    [ USDT BEP20 ▼ ]

    TX Hash:
    [___________]

    [ Submit & Verify ]

    Verification Result:
    Valid / Invalid / Already Used
    -----------------------------------

No other functionality is required in the peer panel.

## 2.3 Normal User

Normal users must have separate registration and login functionality.

Normal user features:

- Register.
- Log in.
- Log out.
- Access authenticated user panel.
- View available deposit methods.
- Select a deposit method.
- Receive one randomly selected active receiving address for that method.
- Submit transaction information where required.

No email verification is required.

Admin login must use:

- 6-digit numeric PIN only.

Example:

    PIN: 123456

Requirements:

- Exactly 6 digits.
- Numbers only.
- PIN stored as a secure hash.
- Never return PIN hash through an API.
- Secure admin session after successful login.
- Admin logout functionality.
- Protect all admin routes with authentication middleware.
- Add brute-force protection.
- Add temporary lockout after repeated invalid attempts.

Peer login must use:

- Unique Peer ID, username, code, or another admin-created identifier.
- 6-digit numeric PIN.

Example:

Requirements:

- PIN must contain exactly 6 numeric digits.
- Store only a secure hash of the PIN.
- Never expose PIN or PIN hash through an API.
- Peer must be active to log in.
- Disabled peers must be denied access.
- Create a secure session after successful login.
- Peer session must only access peer-specific routes.

# 5. File Safety & Concurrency

Because there is no database, concurrency safety is extremely important.

Implement:

- Atomic file writes.
- Temporary-file replacement.
- File locks.
- TX hash verification locks.
- Duplicate prevention.
- Safe simultaneous requests.

Recommended safe-write flow:

    Read current file
           ↓
    Acquire lock
           ↓
    Re-read latest data
           ↓
    Modify data
           ↓
    Write to temporary file
           ↓
    Atomically replace original file
           ↓
    Release lock

The system must prevent two simultaneous requests from successfully claiming the same TX hash.

# 9. Random Deposit Address Selection

When a normal user selects a deposit method:

1. Backend receives selected method ID.
2. Backend loads active addresses for that method.
3. Backend randomly selects one active address.
4. Backend creates a unique deposit request.
5. Backend permanently records the selected address against that deposit request.
6. Frontend receives the selected address.
7. Frontend displays that address to the user.

Do not randomly select a new address every time the page refreshes for the same active deposit request.

Once an address is assigned to a deposit request, preserve that assignment.

Example deposit request:

    {
      "id": "deposit_123456",
      "userId": "user_001",
      "methodId": "usdt_bep20",
      "assignedAddress": "0x...",
      "status": "awaiting_txhash",
      "createdAt": "2026-07-15T10:00:00Z"
    }

# 10. Peer TX Hash Submission Flow

This is the main and only functional feature of the peer panel.

Flow:

    Peer logs in
          ↓
    Peer opens TX verification panel
          ↓
    Peer selects transaction method
          ↓
    Peer enters TX hash
          ↓
    Backend validates TX hash format
          ↓
    Backend checks global duplicate registry
          ↓
    If already used → Reject
          ↓
    Lock TX hash
          ↓
    Fetch transaction directly from blockchain
          ↓
    Independently verify transaction
          ↓
    Valid → Permanently tie TX hash to peer
          ↓
    Invalid → Reject
          ↓
    Release lock

---

# 11. TX Hash Ownership Rule

Once a valid TX hash is successfully submitted and verified by a peer:

- Permanently tie that TX hash to that peer.
- Save the peer ID with the transaction.
- Save the exact verification timestamp.
- Save all independently fetched blockchain information.
- The same TX hash cannot be tied to another peer.
- The same TX hash cannot be submitted again.
- A different peer attempting to submit it must receive an "Already Used" response.

Example:

    {
      "txHash": "0xabc123...",
      "peerId": "peer_001",
      "methodId": "usdt_bep20",
      "status": "verified",
      "verifiedAt": "2026-07-15T10:30:00Z"
    }

Important:

A TX hash must only be permanently tied to a peer after successful blockchain verification.

An invalid TX hash must not be claimed by a peer.

# 12. Global Duplicate TX Hash Prevention

Create one global registry of consumed/verified TX hashes.

Example:

    {
      "txHash": "0xabc123...",
      "peerId": "peer_001",
      "methodId": "usdt_bep20",
      "transactionId": "transaction_001",
      "consumedAt": "2026-07-15T10:30:00Z"
    }

Before verifying any TX hash:

1. Normalize the TX hash according to network requirements.
2. Search the global consumed registry.
3. If found, immediately reject it.
4. If not found, acquire a lock for that TX hash.
5. Check the registry again after acquiring the lock.
6. Perform blockchain verification.
7. If valid, atomically save the transaction and mark the TX hash as consumed.
8. Release the lock.

This prevents race conditions.

Recommended architecture:

    src/
    └── verifiers/
        ├── bsc/
        │   ├── native-verifier
        │   └── token-verifier
        ├── ethereum/

        │   ├── native-verifier
        │   └── token-verifier
        ├── bitcoin/
        │   └── btc-verifier

Do not create one insecure generic verifier that blindly trusts frontend values.

Each verifier must understand its own blockchain's:

- Transaction structure.
- Address format.
- Token transfer events.
- Contract addresses.
- Transaction status.
- Block confirmations.
- Native coin transfers.
- Token transfers.

# 14. Blockchain Verification Requirements

For every submitted TX hash, independently verify:

- TX hash exists on the blockchain.
- Transaction is confirmed or has the required confirmations.
- Transaction was successful.
- Transaction did not revert or fail.
- Correct network was selected.
- Correct receiving address.
- Correct token contract address where applicable.
- Correct token/coin.
- Actual transferred amount.
- Actual sender address.
- Actual recipient address.
- Block number.
- Confirmation count.

Never trust frontend-provided values for:

- Amount.
- Sender.
- Recipient.
- Token.
- Contract address.
- Network.
- Transaction status.
- Confirmations.

All of these must come directly from blockchain data.

---

# 15. Strict Receiving Address Verification

A valid deposit transaction must be sent to an admin-configured receiving address for the selected method.

Required validation:

    Actual blockchain recipient
              =
    Admin-configured address for selected method

If verification is related to a normal user's specific deposit request, use stricter validation:

    Actual blockchain recipient
              =
    Address assigned to that exact deposit request
              =
    Admin-configured address for that method

Reject if:

- Recipient is an outside address.
- Recipient is not configured by admin.
- Recipient belongs to another method.
- Wrong network was used.
- Wrong token contract was used.

---

# 16. Token Transfer Verification

For token transfers such as USDT:

Do not only check the top-level transaction to field.

The verifier must inspect the actual token transfer event or equivalent blockchain data.

For EVM networks:

- Fetch transaction receipt.
- Confirm receipt status is successful.
- Parse token contract logs.
- Find the correct Transfer event.
- Verify token contract address.
- Extract actual sender.
- Extract actual recipient.
- Extract raw amount.
- Apply token decimals.
- Verify recipient against configured deposit addresses.

This is critical because the top-level transaction recipient may be the token smart contract, not the actual receiving wallet.

# 26. Final Non-Negotiable Rules

1. No database of any kind.
2. Use JSON/text-file-based storage.
3. Normal users can register and log in.
4. No email verification.
5. Admin login uses exactly a 6-digit numeric PIN.
6. Admin creates peers.
7. Peers cannot self-register.
8. Peer login uses a unique identifier plus exactly a 6-digit numeric PIN.
9. Peer panel contains only TX hash submission and verification functionality.
10. A valid TX hash is permanently tied to the peer who successfully submits it first.
11. The same TX hash can never be used twice.
12. Concurrent submissions of the same TX hash must not both succeed.
13. All TX verification must happen directly against blockchain data.
14. Never trust amount, recipient, sender, token, contract, status, or confirmations from frontend input.
15. Transactions sent to outside/unconfigured receiving addresses must be rejected.
16. Token contract must be independently verified for token transfers.
17. Each network/method must have an independent verification module.
18. Admin can add multiple receiving addresses per deposit method.
19. Frontend randomly selects only from active configured addresses.
20. Once assigned to a normal user's deposit request, the receiving address must remain tied to that request.
21. PINs and passwords must never be stored in plain text.
22. No blockchain private key is required for transaction verification.
23. All critical file writes must be atomic.
24. All TX hash claims must use locking to prevent race conditions.
25. Security and correctness are more important than unnecessary complexity.
